# Padrão Criacional Prototype

## Introdução ao Padrão Prototype

O padrão criacional **Prototype** tem como objetivo permitir a criação de novos objetos a partir da cópia de um objeto existente, chamado de protótipo. Em vez de instanciar novos objetos diretamente, o sistema clona um objeto já configurado, reutilizando seu estado inicial.

Esse padrão é definido no catálogo GoF como uma forma de especificar os tipos de objetos a serem criados usando uma instância protótipo e criar novos objetos pela cópia dessa instância.

## Motivação para o uso do Prototype

Em muitos cenários, a criação de objetos pode ser um processo caro em termos de desempenho ou consumo de memória. Quando objetos possuem estruturas complexas ou exigem inicializações custosas, clonar um objeto existente se torna uma alternativa mais eficiente do que criar um novo a partir do zero.

Além disso, o padrão Prototype ajuda a reduzir o acoplamento entre o código cliente e as classes concretas dos objetos que ele cria.

## Conceito Central do Padrão

O conceito principal do Prototype é **copiar um objeto existente**, em vez de criar uma nova instância diretamente. O próprio objeto é responsável por fornecer a lógica de clonagem, delegando esse processo para si mesmo.

Para isso, o padrão define uma interface comum que declara uma operação de clonagem. Todas as classes que desejam ser clonadas devem implementar essa interface.

## Estrutura Conceitual do Padrão

Os principais participantes do padrão Prototype são:

- **Prototype**  
    Declara a interface comum para objetos capazes de clonar a si mesmos.
    
- **Concrete Prototype**  
    Implementa a interface Prototype e define como a clonagem deve ser realizada.
    
- **Client**  
    Cria novos objetos solicitando a clonagem de um protótipo existente, sem conhecer sua classe concreta.
    

## Vantagens do Padrão Prototype

- Elimina a sobrecarga dispendiosa da inicialização de objetos complexos.
    
- Facilita a criação de múltiplos objetos semelhantes com pequenas variações.
    
- Reduz o uso excessivo do operador de criação direta de objetos.
    
- Fornece uma alternativa à herança para criação de objetos pré-configurados.
    
- Permite que o sistema seja independente de como seus objetos são criados.
    

## Desvantagens do Padrão Prototype

- Pode ser difícil de implementar em hierarquias de classes já existentes.
    
- Exige que todas as classes clonáveis implementem corretamente a lógica de clonagem.
    
- Objetos que possuem referências internas complexas podem gerar problemas.
    
- Referências circulares podem dificultar a clonagem correta.
    
- Pode impactar o desempenho se o protótipo for muito pesado ou complexo.
    

## Quando Utilizar o Padrão Prototype

O uso do padrão Prototype é indicado quando:

- A criação de novos objetos é custosa em termos de tempo ou recursos.
    
- É necessário evitar a criação repetitiva de objetos semelhantes.
    
- O sistema precisa ser independente das classes concretas dos objetos criados.
    
- As classes a serem instanciadas são definidas em tempo de execução.
    
- É mais simples clonar um objeto existente do que criar um novo.
    

## Tipos de Clonagem

### Cópia Superficial (Shallow Copy)

Na cópia superficial, apenas o objeto principal é clonado. Os tipos de valor são copiados normalmente, mas os tipos de referência não têm seus objetos internos duplicados. Nesse caso, tanto o objeto original quanto o clone passam a apontar para as mesmas instâncias internas.

Esse tipo de clonagem pode causar efeitos colaterais quando objetos internos são modificados, pois a alteração será refletida em todos os clones que compartilham a mesma referência.

### Cópia Profunda (Deep Copy)

Na cópia profunda, todos os objetos referenciados direta ou indiretamente também são clonados. Isso garante que o objeto clonado seja completamente independente do objeto original.

Esse tipo de clonagem evita compartilhamento indesejado de referências e é mais seguro quando o objeto possui composição complexa.

## Exemplo Conceitual de Uso

Um cenário comum para o uso do padrão Prototype é em jogos, onde personagens com características básicas semelhantes precisam ser criados em grande quantidade. Em vez de criar cada personagem do zero, um protótipo é configurado e clonado diversas vezes, alterando apenas algumas propriedades específicas.

Esse mesmo conceito pode ser aplicado em sistemas que lidam com documentos, configurações, perfis ou qualquer outro tipo de objeto complexo e repetitivo.

## Considerações Finais

O padrão Prototype é uma solução poderosa para cenários em que a criação de objetos é frequente e custosa. Quando bem aplicado, ele melhora o desempenho, reduz o acoplamento e torna o sistema mais flexível.

Entretanto, seu uso exige cuidado especial na implementação da clonagem, principalmente em objetos que possuem referências internas complexas. A escolha entre cópia superficial e cópia profunda deve ser feita com base no comportamento esperado dos objetos clonados.
